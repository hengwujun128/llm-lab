# `node-exec.mjs`：子进程 `spawn` 说明

本文对应 `src/node-exec.mjs` 中「用 `spawn` 执行 `ls -la`、继承终端 IO、按退出码结束进程」的写法，并说明 `spawn`、`cwd`、常用选项的用途与注意点。

---

## 1. 代码在做什么（整体流程）

| 步骤 | 代码位置 | 作用 |
|------|----------|------|
| 引入 API | `import { spawn }` | 使用 Node 的 `child_process.spawn` 创建子进程 |
| 准备命令与目录 | `command`、`cwd` | 要执行的命令字符串，以及子进程的当前工作目录 |
| 拆成「命令 + 参数数组」 | `[cmd, ...args] = command.split(' ')` | 满足 `spawn(command, args)` 的典型入参形式 |
| 创建子进程 | `spawn(cmd, args, options)` | 非阻塞启动子进程，得到 `ChildProcess` |
| 错误 | `child.on('error')` | 捕获**启动失败**（例如找不到可执行文件），不是命令运行返回非零 |
| 结束 | `child.on('close')` | 子进程及其 stdio 已关闭；根据 `code` 决定本进程以何退出码结束 |

要点：这里没有把 `spawn` 包成 Promise，而是用事件驱动；**退出逻辑在 `close` 里用 `process.exit` 显式结束 Node 进程**，避免子进程结束后脚本仍挂住（若顶层无其它任务）。

---

## 2. `spawn` 是干什么的

### 2.1 定义（在 Node 里）

`spawn` 用于**启动一个新的子进程**，并返回一个 `ChildProcess` 对象。与一次性跑完命令并缓冲全部输出的 `exec` / `execFile` 不同，`spawn` 更偏向：

- **流式**处理 stdout / stderr（默认是 Stream，可配置 `stdio`）；
- **长时间运行**或输出很大的命令（不必等全部输出进内存）；
- 需要精细控制子进程生命周期（监听 `exit` / `close` / `error` 等）。

### 2.2 典型使用场景

- 跑 CLI 工具、构建脚本、测试 runner，且希望**实时**看到输出。
- 子进程**持续输出**或运行较久（日志、开发服务器等）。
- 需要分别处理 stdout/stderr 管道，或后续自己做重定向、解析。

不太适合：只执行一条短命令且只关心最终字符串结果——此时用 `exec` / `execFile`（或 `promisify(exec)`）往往更简单。

### 2.3 使用 `spawn` 时要注意什么

1. **参数形态**：`spawn(command, args, options)` 中 `args` 必须是**字符串数组**；不要把整句命令只塞在第一个参数里（除非配合 `shell` 并清楚语义）。
2. **与 `shell: true` 的关系**：见下文「为什么这么写」；开启 shell 会改变解析方式，且**不要**把未校验的用户输入直接拼进命令。
3. **错误分两种**：
   - `error` 事件：**没能启动**进程（路径错误、权限等）；
   - 退出码非 0：进程**已启动**但命令失败，要在 `close` / `exit` 里看 `code`。
4. **字符串拆分**：`command.split(' ')` 在路径或参数含**空格**时会拆错；生产里更稳妥的是固定 `cmd` + `args` 数组，或使用能正确解析的库。
5. **资源**：子进程结束后注意是否需要取消监听、销毁流（本示例较简单，一般 `close` 后即退出）。

## 3. `cwd` 是什么

### 3.1 含义

`cwd` 来自 `process.cwd()`，表示**当前 Node 进程的工作目录**（一般是你启动脚本时终端所在的目录，或工具设定的目录）。

传给 `spawn` 的选项里的 `cwd` 表示：**子进程启动时的工作目录**（相当于在 shell 里先 `cd` 到该目录再执行命令）。

### 3.2 典型使用场景

- 希望 `ls`、构建工具、Git 等**相对当前项目根目录**执行。
- 单仓库里要在**子包目录**下跑命令（例如 `packages/foo`）。
- 与「脚本文件所在目录」区分：`cwd` 不一定是 `__dirname`，除非你显式设成 `import.meta.dirname` 等。

### 3.3 注意什么

1. **路径必须存在且可读**，否则子进程行为可能异常或立即失败。
2. **安全与可移植性**：不要把不可信输入直接拼进 `cwd`；跨平台注意路径分隔符（Windows vs POSIX）。
3. **与 `shell: true` 无关**：`cwd` 只决定工作目录，不代替「用 shell 解析一整条命令字符串」的需求；两者可同时使用。

## 4. 为什么这样写 `spawn` 的第三个参数（目的与意义）

```javascript
const child = spawn(cmd, args, {
  cwd,
  stdio: 'inherit', // 实时输出到控制台
  shell: true,
})
```

### 4.1 `cwd`

**目的**：让子进程在「与当前 Node 进程相同的工作目录」下执行 `ls -la`，这样列目录的结果与你在同一目录下手动执行 `ls -la` 一致。

**意义**：可重复、可预期；若省略 `cwd`，默认也是父进程的 `cwd`，此处显式写出有利于阅读和维护（也方便以后改成项目子目录）。

### 4.2 `stdio: 'inherit'`

**目的**：子进程的**标准输入、标准输出、标准错误**直接继承父进程（当前终端）。

**意义**：

- `ls` 的输出**实时**出现在终端，而不是缓存在 Node 内存里；
- 交互式或大量输出时体验更好。

若用默认管道，你需要 `child.stdout.on('data', ...)` 手动打印或拼接，本示例选择「直接继承」更简单。

### 4.3 `shell: true`

**目的**：在 Unix 上通常通过 `/bin/sh` 等执行；在 Windows 上会用 `cmd.exe`。**统一「像在一行 shell 里执行」的行为**，并支持某些依赖 shell 的用法（具体仍以 Node 版本与平台为准）。

**意义**：跨平台脚本里常见；与「只传 `cmd` + `args`、不用 shell」相比，有时更省事。

**注意**：`shell: true` 会提高**命令注入**风险——**永远不要**把未转义、未校验的用户输入拼进 `command`。本示例 `command` 为固定字符串，风险低。

### 4.4 与 `cmd` / `args` 的配合

当前写法把 `'ls -la'` 拆成 `cmd = 'ls'`、`args = ['-la']`。在 `shell: true` 下，Node 仍会按平台规则组装子进程；若将来改成带空格的路径或复杂管道，应避免仅靠 `split(' ')`，改为明确的数组或更安全的解析方式。

## 5. 事件处理：`error` 与 `close`

- **`error`**：子进程**未能成功创建**时触发；代码里把 `error.message` 存到 `errorMsg`，在随后非零退出时打印，便于区分「启动失败」与「命令返回错误码」。
- **`close`**：stdio 流已关闭，并给出 `code`（退出码）。`code === 0` 视为成功，`process.exit(0)`；否则带上 `errorMsg`（若有）并 `process.exit(code || 1)`。

---

## 6. 小结表

| 概念 | 一句话 | 何时用 | 主要注意 |
|------|--------|--------|----------|
| `spawn` | 创建子进程，偏流式与长运行 | 实时输出、大输出、要管生命周期 | 区分启动失败与退出码；`args` 须为数组 |
| `cwd` | 子进程工作目录 | 要在指定目录下执行命令 | 路径有效、安全、跨平台 |
| `stdio: 'inherit'` | 子进程 IO 接到当前终端 | 希望像手动敲命令一样看输出 | 不利于在程序里解析输出（需改管道） |
| `shell: true` | 经系统 shell 执行 | 需要 shell 行为或跨平台习惯写法 | 严防未校验的用户输入 |
