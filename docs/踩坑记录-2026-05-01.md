# 踩坑记录 2026-05-01

## 1. MySQL Docker 容器连接密码错误

**问题**：代码中 `password: 'admin'`，实际容器密码不是这个，导致 `Access denied for user 'root'@'localhost'`。

**排查**：
```bash
docker inspect ruoyi-mysql --format '{{range .Config.Env}}{{println .}}{{end}}' | grep -i mysql
```

**解决**：从容器环境变量中找到正确密码 `123456`，修改连接配置。

**经验**：Docker 容器密码从 `MYSQL_ROOT_PASSWORD` 环境变量读取，不要猜测，直接 `docker inspect` 确认。

## 2. 文件中存在 NBSP（不间断空格 `\xa0`）导致 SQL 解析失败

**问题**：从富文本/AI 生成的代码复制粘贴到编辑器，缩进用的是 `\xa0`（Unicode 不间断空格）而非普通空格，MySQL 解析器无法识别，SQL 语句报语法错误。

**错误信息**：
```
ER_PARSE_ERROR: You have an error in your SQL syntax
```

**排查**：
```python
with open('file.mjs', 'r') as f:
    lines = f.readlines()
for line in lines:
    print(repr(line))  # 会看到 \xa0 字符
```

**解决**：
```python
content = content.replace('\xa0', ' ')
```

或 VS Code 中用正则查找替换：查找 `\xa0`，替换为普通空格。

**附加问题**：`multipleStatements: true` 模式下，SQL 行内注释 `-- xxx` 会导致后续语句解析异常，建议移除行内注释。

**预防**：`.editorconfig` 添加如下配置，防止非标准空格混入：
```ini
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true
```

> Prettier 不会处理字符串内部的 `\xa0`，需手动或脚本处理。

## 3. LangChain `withStructuredOutput` + `functionCalling` 不支持顶层 array schema

**问题**：使用 `z.array(schema)` 作为 `withStructuredOutput` 的 schema，同时 `method: 'functionCalling'`，LLM 返回 `{ "items": [...] }` 但解析器用 array schema 校验，报类型不匹配错误。

**错误信息**：
```
OutputParserException: Failed to parse.
Error: [{"expected":"array","code":"invalid_type","path":[],"message":"Invalid input: expected array, received object"}]
```

**根本原因**：OpenAI Function Calling 协议要求参数 JSON Schema 顶层必须是 `object`，不能是 `array`。LangChain 内部自动将 `z.array()` 包裹为 `{ items: [...] }`，但解析时仍用原始 array schema 校验，导致不匹配。

**解决**：将 schema 手动包裹一层 object：
```js
// ❌ 错误
const schema = z.array(itemSchema)

// ✅ 正确
const schema = z.object({
  items: z.array(itemSchema).describe('数组描述'),
})

// 使用时解构取值
const { items: results } = await structuredModel.invoke(prompt)
```

**各 method 对比**：

| method | DashScope/Qwen 支持 | 顶层 array |
|---|---|---|
| `functionCalling` | ✅ | ❌ 需包 object |
| `jsonMode` | ✅ | ❌ 需包 object |
| `jsonSchema` | ❌ | ✅ |

> `jsonSchema` 仅 OpenAI gpt-4o 及以上原生模型支持，DashScope/Qwen 不支持。

## 4. pnpm catalog 引用但未定义导致包安装缺失

**问题**：`package.json` 中写了 `"mysql2": "catalog:"` 和 `"zod-to-json-schema": "catalog:"`，但 `pnpm-workspace.yaml` 的 `catalog:` 块中没有对应条目，导致 pnpm 跳过安装，运行时报 `Cannot find package`。

**错误信息**：
```
Error [ERR_MODULE_NOT_FOUND]: Cannot find package 'zod-to-json-schema'
ERR_PNPM_CATALOG_ENTRY_NOT_FOUND_FOR_SPEC: No catalog entry 'mysql2' was found for catalog 'default'.
```

**解决**：在 `pnpm-workspace.yaml` 补充缺失的 catalog 条目：
```yaml
catalog:
  mysql2: ^3.19.1
  zod-to-json-schema: ^3.25.2
```

然后重新执行 `pnpm install`。

**经验**：在 monorepo 中新增 `catalog:` 引用时，必须同步在 `pnpm-workspace.yaml` 的 `catalog:` 块中定义版本号，否则安装时会静默跳过（或报错），但包可能因为被其他包间接依赖而存在于 `.pnpm` 目录，造成"明明装了却找不到"的假象。
