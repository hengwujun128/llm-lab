# Monorepo + pnpm 最佳实践

## 背景与问题

在 Monorepo 里，常见诉求是“依赖统一管理、避免重复安装、子项目可共享使用”。  
实践中通常会出现两种实现路径：

1. **外层统一依赖 + 子包不声明 + hoist 暴露**
2. **根 catalog 统一版本 + 子包保留依赖声明（推荐）**

本仓库的实际问题是：`memory-test` 使用了 `@zilliz/milvus2-sdk-node`，但依赖只在其它子项目里声明，导致解析失败。  
这个问题本质上不是“有没有安装”，而是**依赖边界与解析规则不一致**。

---

## 先说结论

**业界更推荐：根 catalog 统一版本 + 子包保留声明 + 严格解析（不全局 hoist）**。

原因是它同时满足：

- 版本一致性（由 catalog 统一）
- 依赖边界清晰（谁用谁声明）
- 环境一致性（本地/CI/容器行为更一致）
- 可迁移性（子包可独立运行、独立发布、独立测试）

---

## 两种方案的本质区别

### 方案 A：外层统一 + 子包不声明 + hoist

做法：

- 把依赖都放在根 `package.json`
- 子包直接 `import`，但不声明依赖
- 借助 `shamefully-hoist=true` 让依赖被“抬平”到更容易被找到的位置

优点：

- 上手快，改动少
- 适合短期实验、PoC、一次性脚本
- 依赖都在根部，看起来集中

缺点：

- 依赖关系是“隐式”的，边界不清晰
- 子包可移植性差（单独拿出去经常缺依赖）
- 容易出现“本地能跑、CI 不稳”的情况
- 后续重构/拆包/发布成本高

---

### 方案 B：根 catalog 统一版本 + 子包保留声明（推荐）

做法：

- 每个子包在自己的 `package.json` 显式声明依赖
- 版本号用 `catalog:`，由根 `pnpm-workspace.yaml` 统一维护
- 保持严格解析（`shamefully-hoist=false`）

优点：

- 依赖边界清楚：看子包 manifest 就知道依赖面
- 版本统一：升级时改 catalog 即可
- CI 稳定：减少幽灵依赖（phantom dependency）
- 更符合 pnpm 默认设计理念与团队协作要求

代价：

- 需要在子包写依赖声明（看起来有少量重复）
- 新增依赖时要改“对应子包 + catalog”（规范流程）

---

## 为什么“子包声明 + catalog 统一”是最佳平衡

Monorepo 有两个维度，不能混为一谈：

- **依赖归属（ownership）**：谁在用，谁声明
- **版本策略（version policy）**：版本如何统一管理

方案 B 把两件事拆开处理：

- 子包声明解决“归属清晰”
- catalog 解决“版本统一”

这正是可维护性最强的组合。

---

## 本仓库落地建议（可直接执行）

### 1) 根 `pnpm-workspace.yaml`

- 维护 `packages` 工作区范围
- 维护 `catalog` 统一版本（如 `@langchain/*`、`@zilliz/*`、`dotenv` 等）

### 2) 子包 `package.json`

- 每个子包保留自己的 `dependencies/devDependencies`
- 依赖版本写 `catalog:`（或明确 semver，按团队约定）
- 禁止“只在根装、子包不声明却直接 import”

### 3) `.npmrc`

- 建议保持：
  - `shamefully-hoist=false`
  - `auto-install-peers=true`（可按团队偏好调整）
  - `strict-peer-dependencies=false/true` 按项目容忍度决定

### 4) 安装与验证

- 开发机：`pnpm install`
- CI：`pnpm install --frozen-lockfile`
- 关键包可用 `pnpm --filter <pkg> exec node -e "import('xxx')"` 做快速冒烟

## 常见误区与解释

### 误区 1：根安装了，子包就一定能用

不一定。pnpm 默认是严格解析，不鼓励隐式穿透依赖。  
即使某些时候“恰好可用”，也可能在 CI、换机器、换 Node 版本时失效。

### 误区 2：去掉子包依赖声明能减少维护成本

短期减少，长期增加。  
一旦拆分包、迁移代码或做按包测试，隐式依赖会集中爆雷。

### 误区 3：hoist 就是最佳实践

hoist 是兼容手段，不是默认最佳实践。  
它可以解决某些历史包或工具链问题，但会削弱边界约束。

---

## 什么时候可以接受“外层统一 + hoist”

可以，但建议明确只用于以下场景：

- 纯个人实验仓库
- 生命周期短（几天/几周）
- 不会拆包、不会发布、不会严格 CI
- 团队对依赖边界不敏感

一旦项目进入“多人协作、持续维护、需要稳定交付”，应切回方案 B。

---

## 业界标准实践清单（Checklist）

- [ ] 子包显式声明自身依赖（谁用谁声明）
- [ ] 版本统一走 `catalog`
- [ ] 默认严格解析（不全局 hoist）
- [ ] CI 使用 `--frozen-lockfile`
- [ ] 定期检查“import 与声明是否一致”
- [ ] 对 peer dependency 规则有明确策略（严格或容忍）

---

## 推荐团队规范（可写入 README / CONTRIBUTING）

1. 新增依赖时，先判断归属子包，再在该子包声明。
2. 若是共享版本依赖，优先写 `catalog:`，并在 `pnpm-workspace.yaml` 维护版本。
3. 不允许通过 hoist 解决“忘记声明依赖”问题。
4. PR 合并前至少做一次按包验证（`pnpm --filter <pkg> ...`）。
5. CI 保持锁文件一致性检查，避免隐式漂移。

---

## 一句话总结

**Monorepo 下最稳妥的方式是：子包声明依赖边界，根 catalog 统一版本策略，pnpm 保持严格解析。**  
这样既能共享版本治理收益，也不会牺牲工程可维护性与可预测性。
